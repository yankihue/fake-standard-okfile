Repository Dossier

Complete project inventory

The following artifacts constitute the full delivery of RFC 0001. Each is version-controlled, reviewable, and machine-verifiable.

RFC-0001-OKFILE.md
Normative specification. Defines the Conformance Aperture, byte-level grammar, diagnostic code taxonomy, security considerations, IANA-style token registry, and implementation mandates. 310 lines of RFC 2119 language.
okfile/conformance.py
Packaged reference implementation. Accepts a file path, reads raw bytes, classifies the candidate against the canonical sequence 4f 4b 0a, and emits a structured diagnostic code. Supports --ceremony and --format flags (output: text, json, certificate) for integration into CI pipelines through the okfile console script.
Diagnostic Code System
Nine-category classification: OKF000 (conforms), OKF001 (empty), OKF002 (missing LF), OKF003 (CRLF), OKF004 (BOM), OKF005 (length mismatch), OKF006 (byte mismatch), OKF007 (provenance phrase missing), OKF008 (I/O or manifest error). Each code carries a deterministic cause and a prescribed remediation.
fixtures/vectors.json
Structured fixture manifest. Declares 11 test vectors, each with name, hex, valid, and code fields. Serves as the single source of truth for conformance testing.
tests/test_okfile_conformance.py
Automated test suite. Reads the fixture manifest, executes the conformance CLI against every declared candidate, and asserts diagnostic code parity. Designed for Python's unittest framework with no external dependencies.
docs/IMPLEMENTATION-GUIDE.md
Prose implementation guide. Covers file I/O byte discipline, error-code interpretation, integration patterns for build systems and deployment pipelines, and guidance for extending the diagnostic taxonomy without narrowing the Conformance Aperture.
conformance-lab/
Self-contained conformance laboratory. Provides a reproducible environment for executing the full test suite, generating conformance certificates, and auditing candidate artifacts against the specification.
Repository & Deployment
Prepared as a public GitHub repository with package metadata, CI, MIT license, and static site configuration. The dossier can be served from site/ with no runtime dependencies.

The Standard

One accepted sequence. Zero interpretive latitude.

An OK file is a finite byte sequence of length exactly three. Consumers MUST read candidates as raw bytes and MUST accept only the sequence 0x4f 0x4b 0x0a — corresponding to the ASCII characters O, K, and a line feed.

Producers MUST NOT localize, lowercase, titlecase, decorate, quote, compress, encrypt, trim, or forgive. The Conformance Aperture contains exactly one sequence. Anything outside the aperture is not nearly OK, culturally OK, or OK after normalization. It is invalid.

This standard treats three bytes as if they were public infrastructure. The format is intentionally trivial. The rigor is not.

Canonical Sequence
  1. 0x4FO
  2. 0x4BK
  3. 0x0ALF

All other sequences are rejected without ceremony.

  --ceremony \
  --format certificate \

Repository & Deployment

Source control and production infrastructure

Public GitHub Repository

All artifacts — specification, source code, fixtures, tests, documentation, package metadata, CI, license, and site assets — are version-controlled for public release.

Vercel Deployment

The public interface is deployed to Vercel via vercel.json. Clean URLs, no trailing slashes, output directory site/. The deployment serves this dossier and the browser conformance interface as a static site with zero runtime dependencies.

Conformance Laboratory

The conformance-lab/ directory provides a reproducible verification environment. Any implementer may clone the repository, execute the test suite, and produce a machine-verifiable conformance certificate asserting compliance with RFC 0001.
